# 什么时候不该用 Test Agents——4 种场景宁可手写

> 不是所有场景都适合 Playwright Test Agents。
> 强行用 = 浪费时间、引入风险、给团队埋雷。
> 这份清单贴在团队 wiki，PR 上引用它来说"这次我们不用 agent"。

---

## 场景 1：高风险关键业务

**典型业务**：支付、退款、医疗诊断、风控决策、合规审计

**为什么不该用**：
- 这些业务对断言**完备性**要求极高，错一个边界就是生产事故
- AI 生成断言时倾向"看起来对"，不倾向"穷举边界"
- Healer 自愈一旦把真 bug 当 locator 失效"修没了"，后果是钱、命、合规

**该怎么做**：
- 这类用例**手写**，由资深测试 + 业务专家结对评审
- AI 只允许做"测试点辅助分析"，不允许直接生成 spec
- 完成的手写 spec 进 CI 用 `--repeat-each=3` 跑 3 遍验证稳定

---

## 场景 2：页面没有稳定语义

**典型场景**：第三方组件库 SSR、Canvas/SVG 绘制的图表、嵌套 iframe、富文本编辑器

**为什么不该用**：
- Planner 探索阶段拿不到稳定 ARIA / label / text
- Generator 写出来的 locator 只能用 CSS 路径——回到了"录制脚本"的老路
- Healer 修起来更脆弱

**该怎么做**：
- **先治理可访问性**，再用 Test Agents
- 推动前端给关键元素加 `aria-label` / `role` / `data-testid`
- 治理前用传统手写 + 截图回归

---

## 场景 3：缺少 seed 文档 / 无人维护业务规则

**典型场景**：祖传系统没有 PRD、产品同学拒绝写规则、需求只在群聊和会议里

**为什么不该用**：
- 没有 seed 的 Planner 只能在 UI 表面打转，发现不了业务 bug
- 没有 seed 的 Generator 写出来的 spec 全是页面级断言，进 CI 也保护不了业务
- 你越用 AI，团队越懒得写规则，恶性循环

**该怎么做**：
- **先补 seed**，再用 Test Agents
- 用 Planner 反过来逼问产品：这里没规则我没法测——倒逼业务方梳理
- seed 维护责任写进 Definition of Done

---

## 场景 4：团队没有 review 文化

**典型表现**：PR 秒批、测试代码没人看、断言写啥都过

**为什么不该用**：
- AI 生成的 spec 必须人审，没人审等于把 AI 的偷工减料直接进 CI
- Healer 的灾难性修复必须人审，没人审等于把 bug 直接放生
- 用 Test Agents 反而**放大**团队的 review 缺失

**该怎么做**：
- **先建立 review 习惯**，再引入 AI
- 用 ep05 的 review 清单做 PR 模板，硬性要求
- 至少 1 名资深测试参与每次 spec PR review

## 自查决策表

回答 4 个问题，全是"是"再用 Test Agents：

| 问题 | 你的答案 |
|---|---|
| 这个业务不属于支付/医疗/风控等高风险类别？ | [ ] |
| 页面元素有稳定的 ARIA role / label / testid？ | [ ] |
| 我们有持续维护的业务规则文档（seed.md 或等价物）？ | [ ] |
| 团队 PR 至少有 1 人 review 测试代码？ | [ ] |

任何一项是"否"——先解决那项，再用 Test Agents。

---

## 灰色地带：可以"部分"用 Test Agents 的场景

| 场景 | 可用部分 | 不可用部分 |
|---|---|---|
| 支付前的购物车流程 | Planner / Generator | 支付确认页坚持手写 |
| 内嵌图表 dashboard | 周边表单和筛选用 Planner | 图表本身用截图回归 |
| 祖传系统改造 | 先用 Planner 帮你"理解现状" | 不让 Generator 立刻生成 spec |
| 跨多个第三方 SaaS 集成 | 用 Healer 维护内部页 | 第三方页面坚持手写或 stub |

---

## 一句话总结

> **Test Agents 的核心价值是"AI 给你打草稿"**。
> 草稿能不能用，取决于你的业务能不能承受"草稿写错"的代价。
