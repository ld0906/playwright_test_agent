守门 prompt · 把规矩写死

PROMPT healer-prompt.md 3 段守门规则 
你扮演 Healer Agent。tests-agents/specs/ 有用例失败...

【允许的修复类型】
1. 按钮/链接文案改名
2. 元素位置变化但语义相同
3. 多语言切换导致的文案变化

【禁止的修复类型（必须停手）】
1. form input 的 name 属性变化
2. 涉及 API URL / 字段名 / 响应结构变化
3. 多个用例同时失败
4. 你需要修改 assertion 才能让用例通过

【输出要求】
- 列出每个失败用例
- 标记 [AUTO-FIX] 或 [NEEDS-HUMAN]
- AUTO-FIX 的直接改，给 diff
- NEEDS-HUMAN 的不改，写明原因和建议
✓ ALLOW
  • 按钮 / 链接 / 标题文案改名
  • 元素位置变化，语义未变
  • 多语言切换
✗ DENY · 4 类必须停手
  • form input name 变化（后端契约）
  • API URL / 字段 / 结构变化
  • 多用例同时挂（可能大改）
  • 需要改 assertion 才能通过
[NEEDS-HUMAN]auth.spec.ts TC-01
失败原因：input[name=username] → userId
这是 form field name 变化，可能影响后端契约。
请人工确认后端 API 字段是否同步。
CORE守门规则不是事后 review，是事前 prompt。写 prompt 多花 5 分钟，省几小时事故。
